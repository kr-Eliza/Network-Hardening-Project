## J+30 — Correction des risques critiques (R-01, R-02, R-03)

### Certificat TLS (R-01)

- [ ] Remplacer le certificat auto-signé par un certificat Let's Encrypt via `certbot`
  ```bash
  apt install certbot python3-certbot-nginx
  certbot --nginx -d td4.example.com
  ```
- [ ] Configurer le renouvellement automatique via cron ou systemd timer
  ```bash
  systemctl enable certbot.timer
  ```
- [ ] Porter le HSTS `max-age` à 31536000 dans nginx_final.conf
- [ ] Ajouter `ssl_stapling on;` et `ssl_stapling_verify on;` une fois le certificat CA valide

### PKI IPsec (R-02)

- [ ] Créer une CA strongSwan dédiée
  ```bash
  pki --gen --type rsa --size 4096 --outform pem > ca.key.pem
  pki --self --ca --lifetime 3650 --in ca.key.pem --type rsa \
      --dn "CN=VPN CA TD5" --outform pem > ca.cert.pem
  ```
- [ ] Émettre un certificat par gateway (gw-fw, gw-fwB)
- [ ] Mettre à jour ipsec.conf : `leftcert=gw-fw.cert.pem`, `rightcert=gw-fwB.cert.pem`
- [ ] Supprimer `/etc/ipsec.secrets` après migration réussie
- [ ] Documenter la procédure de révocation (CRL)

### Suricata IPS (R-03)

- [ ] Tester le passage en mode IPS en environnement de staging
  ```bash
  iptables -I FORWARD -j NFQUEUE --queue-num 0
  suricata -c /etc/suricata/suricata.yaml -q 0
  ```
- [ ] Mesurer l'impact sur les performances (latence, throughput) avec `iperf3`
- [ ] Définir une politique de blocage pour les règles à haute confiance uniquement
- [ ] Conserver les règles locales (SID 9000001–9000003) en mode `drop` au lieu de `alert`

## J+60 — Renforcement de la détection et de l'observabilité (R-04, R-05, R-10)

### Centralisation des logs (R-04)

- [ ] Déployer Elasticsearch + Kibana sur une machine dédiée (ou conteneur Docker)
- [ ] Installer Filebeat sur `sensor-ids`, `srv-web`, `gw-fw`
  ```bash
  apt install filebeat
  filebeat modules enable suricata nginx system
  filebeat setup && systemctl start filebeat
  ```
- [ ] Configurer des dashboards Kibana pour :
  - Alertes Suricata par SID et par heure
  - Top sources IP générant des alertes
  - Erreurs nginx 4xx/5xx groupées par endpoint
  - Connexions SSH refusées (journal sshd)
- [ ] Configurer des alertes email/Slack sur les SIDs critiques (9000001, 9000002, 9000003)

### Anti-brute-force SSH (R-05)

- [ ] Installer fail2ban sur srv-web
  ```bash
  apt install fail2ban
  ```
- [ ] Configurer `/etc/fail2ban/jail.d/sshd.conf` :
  ```ini
  [sshd]
  enabled  = true
  port     = 22
  filter   = sshd
  logpath  = /var/log/auth.log
  maxretry = 5
  findtime = 600
  bantime  = 3600
  ```
- [ ] Tester le bannissement et la levée de ban manuels
- [ ] Activer également une jail pour nginx (protection contre le scanning)

### Règles IDS (R-10)

- [ ] Installer `suricata-update` et activer ET-Open
  ```bash
  pip install suricata-update
  suricata-update update-sources
  suricata-update enable-source et/open
  suricata-update
  ```
- [ ] Activer uniquement les catégories pertinentes : `exploit`, `malware`, `policy`, `scan`
- [ ] Mesurer le taux de faux positifs pendant 1 semaine avant d'activer des actions de blocage
- [ ] Automatiser la mise à jour hebdomadaire des règles via cron

## J+90 — Maturité opérationnelle et automatisation

### Infrastructure as Code

- [ ] Convertir toutes les configurations en rôles Ansible :
  - `role/nftables` : déploiement et gestion des règles
  - `role/nginx_tls` : configuration nginx + certificats
  - `role/suricata` : installation, règles, mode IDS/IPS
  - `role/strongswan` : tunnel IPsec PKI
  - `role/sshd_hardening` : configuration sshd + clés
- [ ] Versionner l'infrastructure dans un dépôt Git avec branches `dev`, `staging`, `prod`
- [ ] Implémenter un pipeline CI/CD (GitHub Actions ou GitLab CI) :
  1. Validation des configs (nginx -t, nft -c, suricata -T)
  2. Tests de régression R1–R4 automatisés sur une VM éphémère
  3. Déploiement en production après validation

### Revue de sécurité continue

- [ ] Planifier une revue trimestrielle du `Risk_Register.md`
- [ ] Abonner les alertes CVE sur les composants utilisés (nginx, strongSwan, Suricata, OpenSSL)
  - Flux RSS CVE : https://nvd.nist.gov/feeds/json/cve/1.1/nvdcve-1.1-recent.json.gz
  - Alertes Debian Security : https://www.debian.org/security/
- [ ] Effectuer un scan de vulnérabilités annuel (OpenVAS, Nessus, ou testssl.sh complet)
- [ ] Documenter un runbook opérationnel pour chaque composant :
  - Procédure de démarrage/arrêt
  - Procédure de rollback de configuration
  - Procédure de réponse à incident (alerte Suricata critique)

### Tests de non-régression élargis

- [ ] Ajouter un test R5 : vérification de la configuration TLS complète via testssl.sh
- [ ] Ajouter un test R6 : vérification qu'un trafic normal ne génère pas d'alerte IDS (test négatif)
