
# 1. Objective

The objective of this laboratory exercise was to deploy and validate an Intrusion Detection System (IDS) using **Suricata** in a segmented network environment.

The exercise aimed to demonstrate several key IDS engineering tasks:

- verifying network visibility of the IDS sensor
- triggering detection rules deterministically
- creating a custom Suricata detection rule
- validating rule behavior through positive and negative tests
- reducing alert noise through alert tuning
- documenting troubleshooting steps and configuration corrections

The experiment was conducted in a controlled virtual lab environment.

---

# 2. Lab Architecture

The lab environment consists of four virtual machines connected through two network segments: a LAN and a DMZ.

Network topology:

```

client ---- LAN ---- gw-fw ---- DMZ ---- srv-web
|
|
sensor-ids

```

Host configuration:

| Host | Role | IP Address |
|-----|------|------------|
| client | workstation generating traffic | 10.10.10.10 |
| gw-fw | gateway and firewall | 10.10.10.1 / 10.10.20.1 |
| srv-web | web server in the DMZ | 10.10.20.10 |
| sensor-ids | Suricata IDS sensor | 10.10.20.50 |

The IDS sensor monitors traffic on the DMZ network and captures packets exchanged between the firewall and the web server.

The monitored interface was configured in **promiscuous mode**, allowing the IDS to observe traffic not directly addressed to it.

---

# 3. IDS Visibility Verification

Before detection rules can function correctly, it is necessary to verify that the IDS sensor has visibility over network traffic.

Traffic was generated from the client using HTTP and ICMP:

```

curl [http://10.10.20.10/](http://10.10.20.10/)
ping -c 3 10.10.20.10

```

Network packets were captured on the sensor using:

```

sudo tcpdump -i enp0s3 host 10.10.20.10

```

The captured packets confirmed that the sensor was able to observe communication between the client and the DMZ web server.

This step ensured that Suricata had the required network visibility to perform packet inspection.

---

# 4. Community Rule Detection

To validate the IDS detection capability, a deterministic scanning activity was generated using **Nmap**.

Command executed on the client:

```

nmap -sS -sV -p 1-1000 10.10.20.10

```

This scan triggered Suricata community rules designed to detect Nmap scanning activity.

Example alert observed in Suricata logs:

```

ET SCAN Possible Nmap User-Agent Observed
SID: 2024364

```

Multiple alerts were generated because the scan produced repeated HTTP requests containing the Nmap user-agent.

This confirmed that the IDS was correctly inspecting network traffic and detecting reconnaissance activity.

---

# 5. Custom Detection Rule

A custom Suricata rule was implemented to detect HTTP requests targeting the `/admin` path.

Custom rule definition:

```

alert http $HOME_NET any -> $HOME_NET 80 (msg:"TD3 CUSTOM - HTTP request to /admin detected"; flow:to_server,established; http.uri; content:"/admin"; sid:9000001; rev:1; classtype:policy-violation;)

```

This rule inspects HTTP requests sent to the web server and triggers an alert when the request URI contains `/admin`.

The rule demonstrates how Suricata can be extended to detect application-specific behavior.

---

# 6. Rule Validation

The custom rule was validated through both positive and negative tests.

## Positive Test

Command executed:

```

curl [http://10.10.20.10/admin](http://10.10.20.10/admin)

```

Expected result:

```

[1:9000001:1] TD3 CUSTOM - HTTP request to /admin detected

```

The web server returned a **404 Not Found** response because the page does not exist.  
However, the IDS correctly triggered an alert because detection is based on the **HTTP request URI**, not on the server response.

---

## Negative Test

Command executed:

```

curl [http://10.10.20.10/index.html](http://10.10.20.10/index.html)

```

Expected result:

No alert with SID **9000001**.

This confirmed that the rule was specific to `/admin` and did not generate false positives.

---

# 7. Alert Noise and IDS Tuning

During the Nmap scan experiment, the rule **SID 2024364** generated multiple alerts for the same activity.

Repeated alerts can overwhelm analysts and reduce the effectiveness of intrusion detection systems.

To reduce alert noise, a **threshold rule** was implemented.

Threshold configuration:

```

threshold gen_id 1, sig_id 2024364, type limit, track by_src, count 1, seconds 60

```

This configuration limits alerts generated by the rule to **one alert per source IP every 60 seconds**.

---

# 8. Before / After Comparison

The same Nmap scan was executed before and after applying the threshold rule.

Command:

```

nmap -sS -sV -p 1-1000 10.10.20.10

```

Results:

| Stage | Alert Count |
|------|-------------|
| Before tuning | multiple alerts |
| After tuning | significantly reduced alerts |

The tuning configuration successfully reduced duplicate alerts while preserving the detection capability.

This improves the **signal-to-noise ratio** for security analysts.

# 9. Troubleshooting and Configuration Issues

Several configuration issues were encountered during the laboratory and had to be resolved.

## Custom rule file not loaded

Initially, the custom rule did not trigger because the file `local.rules` was not referenced in `suricata.yaml`.

Correction:

```

rule-files:

* suricata.rules
* local.rules

```

---

## Incorrect rule path

Suricata attempted to load the rule file from the wrong directory.

Error observed:

```

No rule files match the pattern /var/lib/suricata/rules/local.rules

```

The correct path used was:

```

/etc/suricata/rules/local.rules

```

---


## Threshold rule placement

Another issue occurred when the threshold configuration was incorrectly placed in `local.rules`.

Since `local.rules` only accepts detection signatures, the threshold rule had to be moved to a dedicated file:

```

/etc/suricata/threshold.config

```

and referenced in:

```

suricata.yaml

```
```

threshold-file: /etc/suricata/threshold.config

```

# 10. Lessons Learned

This laboratory exercise demonstrated several important IDS engineering principles:

- validating IDS sensor visibility before deploying detection rules
- generating deterministic traffic for rule testing
- creating custom Suricata detection signatures
- verifying rule accuracy through positive and negative tests
- reducing alert noise through threshold tuning
- troubleshooting configuration errors during IDS deployment

These practices reflect common workflows used by **Security Operations Center (SOC)** detection engineers.

Through the use of community rules, custom detection signatures, and threshold tuning, the IDS was able to detect suspicious activity while maintaining manageable alert volumes.
